# Full Web3 & MetaMask Integration

This document details a strategic roadmap to fully integrate Ethereum-standard (secp256k1) addresses across the Verimus ledger ecosystem. Currently, Verimus nodes and network tokens are tightly coupled with the native P2P RSA keypairs. This leads to a fragmented Web3 experience where, although the UI binds to a MetaMask account (`web3Account`), the underlying consensus engine and wallet manager still utilize RSA public keys to track balances, identify block creators (`publicKey`), and establish recipients/senders in `TRANSACTION` blocks. This plan bridges the gap between Node Identity and End-User Identity to align completely with industry standards natively.

## User Review Required

> [!WARNING]
> This is a major structural shift. **All historical blocks generated by RSA keys will become incompatible**, requiring a ledger wipe and cold restart of local development nodes upon the roadmap's completion.

### Proposed Solution (Pivoted): Decoupled RSA / EVM Identities
Keep RSA strictly for P2P transport encryption and network overlay topologies while adopting EVM accounts exclusively for layer-2 ledger economics.

**Pros:**
- Complete separation of concerns, capping blast radius.
- Requires virtually zero refactoring to the lower-level routing mechanisms (`p2p/lib/*`).

**Cons:**
- Perpetuates an inherently fragmented architecture requiring persistent bridging logic to relate networking keys with economic wallets.

## Ongoing Verification & Testing Methodology

To hold ourselves accountable, each milestone outlined above will be strictly validated through the following continuous testing approach before advancing to subsequent phases:

### 1. Systematic Automated Testing
- **Execution:** Mandate `npm test` completion prior to closing any Task Breakdown document.
- **Scope:** Unit tests must be updated to track deterministic `ethers.Wallet` mocking abstractions (relieving brittle legacy RSA HTTP stubs). 
- **Hermetic Integrity:** `MongoMemoryServer` test runs must persist completely within isolated environments, confirming local node execution does not write artifacts to disk.

### 2. Manual UI Integration (Chrome)
- **Execution:** Following major backend milestones (specifically Milestones 3 and 4), developers must spin up nodes using `pkill -f tsx; pkill -f nodemon; pkill -f mongo; ./scripts/spawn_nodes.sh --mongo` and execute localized verification within Chrome.
- **Boundaries Checked:** 
  - Verification that transactions appropriately fail or halt if the injected MetaMask prompt is absent or rejected.
  - Ensuring blocks visually propagate to the UI Ledger upon successful signed JSON-RPC execution.
